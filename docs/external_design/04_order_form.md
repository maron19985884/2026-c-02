# 外部設計書（機能仕様書）
## 画面4：注文フォーム

**対応要件:** U-12, U-13, U-14, U-15
**画面パス:** `/checkout`
**Nextコンポーネント:** `src/app/checkout/page.tsx`（クライアントコンポーネント）

---

## 1. 画面概要

注文者情報（氏名・住所・メールアドレス）を入力し、注文内容（カート）を確認したうえで注文を確定する画面。「注文する」ボタン押下で `POST /api/orders` を呼び出し、成功したら注文完了画面へ遷移する。入力ミスはボタン押下時にエラーメッセージで通知する。

## 3. 表示項目と表示条件

### 3-1. ヘッダー（共通）

| 要素 | 表示内容 | 表示条件 |
|---|---|---|
| ロゴ/サイト名 | 書店名テキスト（リンク：`/`） | 常時表示 |
| カートアイコン | カートに入っている総冊数 | 常時表示 |

### 3-2. お届け先情報フォーム

| 要素 | 表示内容 | 表示条件 |
|---|---|---|
| 氏名ラベル | 「氏名 *」（必須マーク付き） | 常時表示 |
| 氏名テキストボックス | 入力値 | 常時表示 |
| 氏名エラー | バリデーションエラーメッセージ | バリデーション失敗時のみ表示（赤文字） |
| 住所ラベル | 「住所 *」（必須マーク付き） | 常時表示 |
| 住所テキストボックス | 入力値 | 常時表示 |
| 住所エラー | バリデーションエラーメッセージ | バリデーション失敗時のみ表示（赤文字） |
| メールアドレスラベル | 「メールアドレス *」（必須マーク付き） | 常時表示 |
| メールテキストボックス | 入力値（`type="email"`） | 常時表示 |
| メールエラー | バリデーションエラーメッセージ | バリデーション失敗時のみ表示（赤文字） |

### 3-3. 注文内容パネル（`OrderSummary` コンポーネント）

| 要素 | 表示内容 | 表示条件 |
|---|---|---|
| 各商品行 | `タイトル × {quantity}冊` と `¥{小計}` | カートアイテムが1件以上あるとき |
| 合計金額 | `合計：¥{totalAmount.toLocaleString()}` | 常時表示 |

### 3-4. 「注文する」ボタン

| 状態 | 表示内容 |
|---|---|
| 通常 | 「注文する」（活性状態） |
| API送信中 | 「送信中...」（非活性・ローディング表示） |

### 3-5. APIエラーメッセージ

| 条件 | 表示内容 |
|---|---|
| APIが 400 を返したとき | `error` フィールドの文言を表示 |
| APIが 404 を返したとき | 「注文の処理に失敗しました。カートを確認してください。」 |
| APIが 500 を返したとき | 「サーバーエラーが発生しました。しばらくしてから再度お試しください。」 |
| ネットワークエラーのとき | 「通信エラーが発生しました。接続を確認してください。」 |

### 3-6. カートが空の場合

| 条件 | 動作 |
|---|---|
| カートが空の状態で `/checkout` に直接アクセス | カート画面（`/cart`）へリダイレクト |

## 4. ユーザー操作と動作

| 操作対象 | 操作 | 動作 |
|---|---|---|
| 各テキストボックス | 文字入力 | 入力値を `useState` で管理。リアルタイムバリデーションは行わない（送信時のみ） |
| 「注文する」ボタン | クリック | 下記「注文確定フロー」を実行 |
| ロゴ/サイト名（ヘッダー） | クリック | 商品一覧画面（`/`）へ遷移 |

**注文確定フロー:**

```
1. フロントエンドでバリデーション実行
   → エラーあり：各フィールド直下にエラーメッセージを表示し、処理を中断
   → エラーなし：次へ

2. ボタンを非活性化し「送信中...」に変更

3. POST /api/orders にリクエスト送信
   {
     customerName, address, email,
     items: cartItems.map(i => ({ bookId: i.bookId, quantity: i.quantity }))
   }

4. レスポンスを受け取る
   → 201 Created：
       a. CartContext.clear() でカートを空にする
       b. レスポンス（orderId, totalAmount 等）を Next.js ルーター経由で
          /order-complete に渡す（URLパラメータ or sessionStorage）
       c. /order-complete へ遷移
   → エラー：
       a. ボタンを再活性化
       b. APIエラーメッセージを表示
```

## 5. バリデーションルール

バリデーションは「注文する」ボタン押下時にフロントエンドで実行する（送信前チェック）。

| フィールド | ルール | エラーメッセージ |
|---|---|---|
| 氏名 | 必須（空文字・空白のみ不可） | 「氏名を入力してください」 |
| 住所 | 必須（空文字・空白のみ不可） | 「住所を入力してください」 |
| メールアドレス | 必須（空文字・空白のみ不可） | 「メールアドレスを入力してください」 |
| メールアドレス | メール形式チェック（正規表現: `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`） | 「有効なメールアドレスを入力してください」 |

**バリデーション実行タイミング:**
- ボタン押下時に全フィールドを一括チェック
- エラーが1件でもあれば API 送信は行わない
- エラーが複数ある場合は全てのエラーメッセージを同時に表示する

**サーバーサイドバリデーション（バックエンド側）:**
- 上記と同等のバリデーションをバックエンドでも実施し、400 エラーを返す（フロントエンド回避策への対策）

## 6. 使用するAPIエンドポイント

| タイミング | メソッド | パス | 説明 |
|---|---|---|---|
| 「注文する」ボタン押下時 | `POST` | `/api/orders` | 注文確定・保存・注文番号発番 |

**リクエストボディ（送信する値）:**

```json
{
  "customerName": "佐藤 次郎",
  "address": "東京都渋谷区道玄坂1-2-3 サンプルビル4F",
  "email": "jiro.sato@example.com",
  "items": [
    { "bookId": "a1b2c3d4-...", "quantity": 2 },
    { "bookId": "b2c3d4e5-...", "quantity": 1 }
  ]
}
```

**成功レスポンスから使用する値:**

| レスポンスフィールド | 注文完了画面への受け渡し用途 |
|---|---|
| `orderId` | 注文番号として表示 |
| `totalAmount` | 合計金額として表示 |
| `items` | 注文商品一覧として表示 |

## 7. 画面遷移先

| 遷移先 | 条件 |
|---|---|
| 注文完了画面 `/order-complete` | `POST /api/orders` が 201 で返ってきた後 |
| カート画面 `/cart` | カートが空の状態でページにアクセスした場合（リダイレクト） |
| 商品一覧画面 `/` | ヘッダーロゴをクリック |
