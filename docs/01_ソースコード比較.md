## 3. フロントエンド実装比較

### 3-1. ページ構成とルーティング

| 画面 | kazukisugawara2 | keitaKobayashi | satoyuya |
|------|-----------------|----------------|----------|

| 注文完了 | `/app/order/complete/page.tsx` | `/app/complete/page.tsx` | `/app/complete/[orderId]/page.tsx` |

### 3-2. データフェッチ方式

```
kazukisugawara2:
  - 商品一覧・詳細：Server Component（SSR）
  - カート・注文：Client Component（useEffect）
  - SSR用とCSR用でAPIのURLを切り替え（INTERNAL_API_URL / NEXT_PUBLIC_API_URL）
  - `cache: 'no-store'` でキャッシュなし

keitaKobayashi:
  - 商品一覧：Server Component + ISR（revalidate = 60）
  - カート・注文：Client Component
  - docker-compose.yml でサービス間ヘルスチェックあり

satoyuya:
  - 全ページ Client Component（'use client'）
  - 全て useEffect でデータ取得
  - NEXT_PUBLIC_API_URL のみ（サーバーサイドURLなし）
```

### 3-3. カート状態管理

#### kazukisugawara2

```
ファイル: /frontend/src/hooks/useCart.ts
方式: カスタムフック + localStorage
```

- `CartItem = { product_id, title, price, quantity }` ── 商品IDと最小限の情報のみ保持
- `loadCart()` / `saveCart()` でlocalStorageを直接操作
- hydrationフラグ（`initialized`）でSSR/CSRの不整合を防ぐ
- `addItem()` / `updateQuantity()` / `removeItem()` / `clearCart()` を提供

#### keitaKobayashi

```
ファイル: /frontend/src/hooks/useCart.ts（CartContext）
方式: React Context + localStorage
```

- `CartItem = { book: Book, quantity: number }` ── Bookオブジェクト全体を保持
- `CartProvider` でコンテキストを提供
- `useState` でカート状態を管理
- `totalAmount` をコンテキストから直接取得可能
- `CartProvider` が `layout.tsx` で全体をラップ

#### satoyuya

```
ファイル: /frontend/src/context/CartContext.tsx
方式: React Context + useReducer + localStorage
```

- `CartItem = { book: Book, quantity: number }` ── Bookオブジェクト全体を保持
- `CartAction` discriminated union: `ADD | UPDATE | REMOVE | CLEAR | HYDRATE`
- `cartReducer` 関数で全ての状態遷移を管理
- `UPDATE` アクションで `quantity <= 0` になると自動削除
- `totalCount`（合計点数）も提供

**比較ポイント：**

| 観点 | kazukisugawara2 | keitaKobayashi | satoyuya |
|------|-----------------|----------------|----------|
| 状態管理パターン | カスタムフック | Context + useState | Context + useReducer |
| カートに保持するデータ | ID＋必要情報のみ | Bookオブジェクト全体 | Bookオブジェクト全体 |
| localStorage保存 | ○ | ○ | ○ |
| SSR対策 | initializedフラグ | なし | effectベース |
| 数量0時の挙動 | 最小値1に制限 | ボタン非活性で実質1以上 | 自動削除 |

## 4. バックエンド実装比較

### 4-1. エントリーポイントとルーティング

### 4-2. データアクセス層の設計

#### kazukisugawara2（クエリ関数分離パターン）
```
backend/src/
├── db/
│   ├── pool.ts             # コネクションプール
│   ├── productsQuery.ts    # 商品クエリ関数群
│   └── ordersQuery.ts      # 注文クエリ関数群
├── controllers/
│   ├── productsController.ts
│   └── ordersController.ts
└── routes/
    ├── products.ts
    └── orders.ts
```

- DB操作をクエリ関数（`getAllProducts`, `createOrderWithItems`）として分離
- コントローラーで入力検証 → クエリ関数呼び出し → レスポンス整形

#### keitaKobayashi（モデル層パターン）
```
backend/src/
├── db/
│   └── index.ts            # コネクションプール
├── models/
│   ├── bookModel.ts        # Book CRUD
│   └── orderModel.ts       # Order CRUD（トランザクション含む）
├── controllers/
│   ├── bookController.ts
│   └── orderController.ts
└── routes/
    ├── books.ts
    └── orders.ts
```

- モデル層（`findAllBooks`, `createOrder`）がDBアクセスをカプセル化
- コントローラーはモデルを呼ぶだけに専念
- 最もレイヤーが明確に分離されている

#### satoyuya（ルート直書きパターン）
```
backend/src/
├── db.ts                   # コネクションプール
└── routes/
    ├── books.ts            # DB操作をルート内に直接実装
    └── orders.ts           # DB操作をルート内に直接実装
```

- コントローラーやモデル層なし
- ルートハンドラーにDBクエリ・バリデーション・レスポンス生成を全て記述

### 4-3. 注文番号生成ロジック

#### kazukisugawara2

#### keitaKobayashi

```
IDのみで注文を識別
```

#### satoyuya
```typescript
// ORD-YYYYMMDDHHmmss-XXXX 形式（日時＋ランダム4桁）
const order_number = `ORD-${now}-${Math.floor(Math.random() * 10000).toString().padStart(4, '0')}`;
```
- INSERT前に生成して直接挿入

## 8. 技術選択の総評

### kazukisugawara2
- Server Components を積極活用し、ページ初期表示のパフォーマンスを優先
- SSR/CSR の切り替えを意識した二重URL設定
- カートは最小限のデータ（ID＋価格）のみ保持し、localStorage容量効率が高い
- 独自の注文番号フォーマット（ゼロパディングID）

### keitaKobayashi
- レイヤードアーキテクチャが最も整っている（routes → controllers → models → db）
- `ApiResponse<T>` による統一レスポンス形式でフロント・バック間の型契約が明確
- Jest + supertest によるテストが唯一動作する実装
- `INTERNAL_API_URL` 対応と docker-compose ヘルスチェックで本番を意識した構成

### satoyuya
- シンプルな構成（ルート直書き）で実装コストを下げた設計
- `useReducer` による予測可能なカート状態管理
- 日時ベースの注文番号で可読性が高い
- `customer_address` を `TEXT` 型にするなど、長い住所への対応意識
