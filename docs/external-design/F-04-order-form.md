# 注文フォーム機能 外部設計書

## 1. 概要
- **機能ID**: F-04
- **機能名**: 注文フォーム機能
- **対象ユーザー**: カートに1冊以上の書籍を追加し、購入意思を固めたユーザー
- **目的・背景**: 配送先情報（氏名・住所・メールアドレス）を入力させ、カート内容と合わせてサーバーに送信することで注文を確定する。入力ミスを防ぐためのバリデーションと、注文内容を見ながら入力できる UI を提供する。
- **作成日**: 2026-05-05
- **バージョン**: 1.1
- **関連要件**: U-12, U-13, U-14, U-15

## 2. 機能概要

`/order` にアクセスすると、カートが空の場合はフルスクリーンで「カートが空です」メッセージと書籍一覧リンクを表示する。カートにアイテムがある場合は、チェックアウトスタイルのヘッダー＋プログレスバー＋2カラムレイアウト（左: 配送先情報フォーム、右: 注文サマリーパネル）を表示する。右サイドにも「注文を確定する」ボタンがあり、フォームの送信ハンドラーを共用する。「注文を確定する」ボタン押下で `POST /orders` を実行し、成功したら注文番号をクエリパラメータとして注文完了画面（`/order/complete`）へリダイレクトする。

## 4. 入出力定義

### 4.1 入力項目

| 項目名 | 型 | 必須 | バリデーション | 説明 |
|---|---|---|---|---|
| name | string | ○ | `!name.trim()` で空チェック | 氏名 |
| address | string | ○ | `!address.trim()` で空チェック | 配送先住所（textarea） |
| email | string | ○ | trim後に `/^[^\s@]+@[^\s@]+\.[^\s@]+$/` | 連絡先メール |

#### バリデーションタイミング
- フォーム送信時のみ（`handleSubmit` 内で `validate()` を呼び出し）
- blur 時の個別バリデーションは未実装

#### バリデーションエラーメッセージ

| 項目 | 条件 | エラーメッセージ |
|---|---|---|
| name | 未入力（trim後空） | 「氏名を入力してください」 |
| address | 未入力（trim後空） | 「住所を入力してください」 |
| email | 未入力（trim後空） | 「メールアドレスを入力してください」 |
| email | 形式不正 | 「メールアドレスの形式が正しくありません」 |

### 4.2 出力項目

#### 画面表示
- 注文サマリー: カート内容（書名・数量・小計）と合計金額・配送料
- バリデーションエラー: 各項目の下に `#CC0000` 文字で表示

#### サーバー送信ペイロード（`POST /orders`）

```typescript
{
  name: string,
  address: string,
  email: string,
  items: [
    {
      book_id: number,   // CartItem.id
      title: string,
      price: number,
      quantity: number
    }
  ]
}
```

## 5. API設計

### 5.1 使用エンドポイント

| メソッド | パス | リクエスト | レスポンス | 説明 |
|---|---|---|---|---|
| POST | `/orders` | `{ name, address, email, items }` | `{ order_number: string }` | 注文を作成 |

#### リクエストボディ例

```json
{
  "name": "山田 太郎",
  "address": "東京都千代田区1-2-3",
  "email": "yamada@example.com",
  "items": [
    { "book_id": 1, "title": "プログラミング入門", "price": 2800, "quantity": 2 },
    { "book_id": 3, "title": "データサイエンス実践", "price": 3800, "quantity": 1 }
  ]
}
```

#### 成功レスポンス例（200）

```json
{ "order_number": "ORD-1714900000000" }
```

#### エラーレスポンス

| ステータス | ボディ | 条件 |
|---|---|---|
| 400 | `{ "error": "Bad Request" }` | 必須項目欠落、items が空配列 |
| 500 | `{ "error": "Internal Server Error" }` | DB接続エラー、トランザクション失敗 |

**注意**: 現在の実装では `res.ok` チェックを行わず直接 `res.json()` を呼んでいる。サーバーエラー時は `data.order_number` が `undefined` になり、`/order/complete?orderNumber=undefined` へ遷移する可能性がある。

### 5.2 新規エンドポイント

不要（既存 `POST /orders` を使用）。

## 6. データ設計

### 6.1 使用テーブル

#### orders テーブル

| カラム | 型 | 値 |
|---|---|---|
| id | INT (PK, AUTO_INCREMENT) | 自動採番 |
| order_number | VARCHAR(30), UNIQUE | `ORD-${Date.now()}` |
| name | VARCHAR(100) | フォーム入力値 |
| address | TEXT | フォーム入力値 |
| email | VARCHAR(255) | フォーム入力値 |
| total_amount | INT | サーバー側で `Σ price × quantity` を計算 |
| created_at | TIMESTAMP | 自動 |

#### order_items テーブル

| カラム | 型 | 値 |
|---|---|---|
| id | INT (PK) | 自動採番 |
| order_id | INT (FK→orders.id) | 直前 INSERT した orders.id |
| book_id | INT | リクエストの `items[].book_id` |
| title | VARCHAR(200) | リクエストの `items[].title` |
| price | INT | リクエストの `items[].price` |
| quantity | INT | リクエストの `items[].quantity` |

### 6.2 データフロー

```
[CartContext.items + フォーム入力]
              ↓ クライアント側バリデーション（validate()）
              ↓ submitting = true、ボタン無効化
              ↓ POST /orders
[Backend: トランザクション開始]
              ↓
[INSERT INTO orders] → orderId 取得
              ↓
[INSERT INTO order_items × N]
              ↓
[COMMIT] → { order_number } 返却
              ↓
[Frontend: clearCart() 実行]
              ↓
[router.push('/order/complete?orderNumber=...')]
```

## 7. ビジネスロジック

### クライアント側処理フロー

1. ページマウント時:
   - `CartContext.items` を読み込み
   - `items.length === 0` の場合は「カートが空です」UI を表示（リダイレクトなし）
2. ユーザーが各フィールドに入力
3. 「注文を確定する」クリック（左フォームまたは右サマリー）:
   1. `validate()` で全項目バリデーション、エラーがあれば `setErrors()` して送信中止
   2. `setSubmitting(true)` でボタン無効化
   3. `items` を `{ book_id: i.id, title: i.title, price: i.price, quantity: i.quantity }` 形式に変換
   4. `POST /orders` を実行（Content-Type: application/json）
   5. 成功時: `clearCart()` を呼び出してカートを空に
   6. `router.push('/order/complete?orderNumber=' + data.order_number)` で遷移
   7. 失敗時（catch）: `setSubmitting(false)` のみ（エラーメッセージ表示なし）

### サーバー側処理フロー

1. リクエストボディを検証（`name`, `address`, `email`, `items` の存在チェック）
2. `total_amount = Σ price × quantity` を計算
3. `order_number = ORD-${Date.now()}` を生成
4. トランザクション開始
5. `orders` テーブルへ INSERT
6. 各 item を `order_items` へ INSERT
7. COMMIT し `{ order_number }` を返却
8. エラー時は ROLLBACK して 500 を返却、`finally` で `conn.release()`

## 8. エラー処理

| エラー条件 | 現在の挙動 |
|---|---|
| 氏名未入力 | 「氏名を入力してください」（フィールド下に赤文字） |
| 住所未入力 | 「住所を入力してください」 |
| メール未入力 | 「メールアドレスを入力してください」 |
| メール形式不正 | 「メールアドレスの形式が正しくありません」 |
| カートが空 | 「カートが空です」＋「書籍一覧へ」リンクを表示 |
| API通信失敗（catch） | `setSubmitting(false)` のみ（エラーメッセージなし） |
| API 400/500 | `res.ok` チェックなし。`data.order_number` が undefined のまま遷移する可能性あり |
| 二重送信 | `submitting = true` 中はボタン `disabled` で防止 |

## 9. 非機能要件

- **パフォーマンス**:
  - 二重送信防止のためボタン無効化を即座に実施
- **セキュリティ**:
  - XSS対策: React の自動エスケープ
  - SQLインジェクション: バックエンドは `mysql2` のパラメータバインディング使用済み
  - サーバー側で `total_amount` を再計算（クライアントの価格を信頼しない）

## 10. 制約・前提条件

- 決済処理はスコープ外。注文確定 = 注文情報の保存のみ。
- 在庫管理スコープ外のため、注文時に在庫数チェックは行わない。
- カートが空の場合、リダイレクトではなく専用UIを表示する。
- 配送料は常に「無料」表示（計算・条件なし）。
- プログレスバーのステップ「2 支払い方法」「3 注文の確定」はUI上の表示のみ（機能なし）。
- フロントエンドコンポーネントは `"use client"` ディレクティブが必須。
- スタイルはインラインスタイルのみ使用。
