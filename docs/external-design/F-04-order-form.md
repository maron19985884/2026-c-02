# 注文フォーム機能 外部設計書

## 1. 概要
- **機能ID**: F-04
- **機能名**: 注文フォーム機能
- **対象ユーザー**: カートに1冊以上の書籍を追加し、購入意思を固めたユーザー
- **目的・背景**: 配送先情報（氏名・住所・メールアドレス）を入力させ、カート内容と合わせてサーバーに送信することで注文を確定する。入力ミスを防ぐためのバリデーションと、注文内容を見ながら入力できる UI を提供する。
- **作成日**: 2026-05-05
- **バージョン**: 1.0
- **関連要件**: U-12, U-13, U-14, U-15

## 2. 機能概要

`/order` にアクセスすると、左カラム（または上部）に配送先情報入力フォーム、右カラム（または下部）にカート内容と合計金額を表示する2カラムレイアウト（モバイルでは縦並び）。氏名・住所・メールアドレスの3項目をすべて必須入力とし、未入力やメール形式不正の場合はインラインでエラーメッセージを表示する。「注文する」ボタン押下で `POST /orders` を実行し、成功したら注文番号をクエリパラメータとして注文完了画面（`/order/complete`）へリダイレクトする。

## 3. 画面・UI設計

### 3.1 画面一覧

| 画面名 | URL/パス | 説明 |
|---|---|---|
| 注文フォーム画面 | `/order` | 配送先情報入力＋注文サマリー表示 |

### 3.2 画面レイアウト

#### 全体構成（PC: 2カラム、モバイル: 縦並び）

```
┌───────────────────────────┬──────────────────────┐
│  配送先情報入力フォーム      │  注文サマリー         │
│                           │                      │
│  氏名:    [        ]      │  ・書籍A x2  ¥6,400  │
│  住所:    [        ]      │  ・書籍B x1  ¥3,200  │
│  メール:  [        ]      │  ─────────────       │
│                           │  合計: ¥9,600         │
│  [ 注文する ]              │                      │
└───────────────────────────┴──────────────────────┘
```

#### フォーム要素

| 項目 | 入力タイプ | プレースホルダ | バリデーション |
|---|---|---|---|
| 氏名 | `<input type="text">` | 「山田 太郎」 | 必須、1〜100文字 |
| 住所 | `<textarea>` または `<input type="text">` | 「東京都〇〇区〇〇1-2-3」 | 必須、1文字以上 |
| メールアドレス | `<input type="email">` | 「example@example.com」 | 必須、メール形式 |

#### 注文サマリー要素

| 要素 | 内容 |
|---|---|
| アイテム一覧 | 書名 × 数量 / 小計 を行ごとに表示 |
| 合計金額 | `total` を強調表示 |

#### ボタン
- 「注文する」: プライマリボタン（背景 `#1e3a8a`、白文字）
  - 送信中は無効化＋「送信中...」表示

#### スタイル仕様
- ページ背景: `#f3f4f6`
- フォーム・サマリーカード: `white`、`box-shadow: 0 2px 8px rgba(0,0,0,0.08)`
- エラーメッセージ: `#ef4444` 系の赤色、フィールド下に表示
- 入力フィールドの枠線: 通常 `#d1d5db`、エラー時 `#ef4444`
- インラインスタイルのみ使用

### 3.3 画面遷移

```
[カート (/cart)]
     │ 「注文手続きへ進む」
     ↓
[注文フォーム (/order)]
     │
     ├─ 入力エラー → 同画面にエラー表示（遷移なし）
     │
     └─ 「注文する」 → POST /orders
            │
            ├─ 成功(200) → [注文完了 (/order/complete?orderNumber=ORD-xxx)]
            │
            └─ 失敗 → 同画面にエラー表示（遷移なし）
```

#### 不正アクセス時
- カートが空の状態で直接 `/order` にアクセスされた場合、`/cart` または `/` にリダイレクトする。

## 4. 入出力定義

### 4.1 入力項目

| 項目名 | 型 | 必須 | バリデーション | 説明 |
|---|---|---|---|---|
| name | string | ○ | 1〜100文字、空白のみ不可 | 氏名 |
| address | string | ○ | 1文字以上（空白のみ不可） | 配送先住所 |
| email | string | ○ | RFC準拠のメールアドレス形式 | 連絡先メール |

#### バリデーションタイミング
- フォーム送信時: 全項目を一括検証
- 推奨: blur イベント時にも個別検証してリアルタイムにフィードバック

#### バリデーションルール詳細

| 項目 | 条件 | エラーメッセージ |
|---|---|---|
| name | 未入力 | 「氏名を入力してください」 |
| name | 100文字超 | 「氏名は100文字以内で入力してください」 |
| address | 未入力 | 「住所を入力してください」 |
| email | 未入力 | 「メールアドレスを入力してください」 |
| email | 形式不正 | 「メールアドレスの形式が正しくありません」 |

### 4.2 出力項目

#### 画面表示
- 注文サマリー: カート内容（書名・数量・小計）と合計金額
- バリデーションエラー: 各項目の下に赤文字で表示

#### サーバー送信ペイロード（`POST /orders`）

```typescript
{
  name: string,
  address: string,
  email: string,
  items: [
    {
      book_id: number,   // CartItem.id
      title: string,
      price: number,
      quantity: number
    }
  ]
}
```

## 5. API設計

### 5.1 使用エンドポイント

| メソッド | パス | リクエスト | レスポンス | 説明 |
|---|---|---|---|---|
| POST | `/orders` | `{ name, address, email, items }` | `{ order_number: string }` | 注文を作成 |

#### リクエストボディ例

```json
{
  "name": "山田 太郎",
  "address": "東京都千代田区1-2-3",
  "email": "yamada@example.com",
  "items": [
    { "book_id": 1, "title": "プログラミング入門", "price": 2800, "quantity": 2 },
    { "book_id": 3, "title": "データサイエンス実践", "price": 3800, "quantity": 1 }
  ]
}
```

#### 成功レスポンス例（200）

```json
{ "order_number": "ORD-1714900000000" }
```

#### エラーレスポンス

| ステータス | ボディ | 条件 |
|---|---|---|
| 400 | `{ "error": "Bad Request" }` | 必須項目欠落、items が空配列 |
| 500 | `{ "error": "Internal Server Error" }` | DB接続エラー、トランザクション失敗 |

### 5.2 新規エンドポイント

不要（既存 `POST /orders` を使用）。

## 6. データ設計

### 6.1 使用テーブル

#### orders テーブル

| カラム | 型 | 値 |
|---|---|---|
| id | INT (PK, AUTO_INCREMENT) | 自動採番 |
| order_number | VARCHAR(30), UNIQUE | `ORD-${Date.now()}` |
| name | VARCHAR(100) | フォーム入力値 |
| address | TEXT | フォーム入力値 |
| email | VARCHAR(255) | フォーム入力値 |
| total_amount | INT | サーバー側で `Σ price × quantity` を計算 |
| created_at | TIMESTAMP | 自動 |

#### order_items テーブル

| カラム | 型 | 値 |
|---|---|---|
| id | INT (PK) | 自動採番 |
| order_id | INT (FK→orders.id) | 直前 INSERT した orders.id |
| book_id | INT | リクエストの `items[].book_id` |
| title | VARCHAR(200) | リクエストの `items[].title` |
| price | INT | リクエストの `items[].price` |
| quantity | INT | リクエストの `items[].quantity` |

トランザクション内で `orders` への INSERT 後、各 item を `order_items` へ INSERT。失敗時はロールバック。

### 6.2 データフロー

```
[CartContext.items + フォーム入力] 
              ↓ クライアント側バリデーション
              ↓ POST /orders
[Backend: トランザクション開始]
              ↓
[INSERT INTO orders] → orderId 取得
              ↓
[INSERT INTO order_items × N]
              ↓
[COMMIT] → { order_number } 返却
              ↓
[Frontend: clearCart() 実行]
              ↓
[router.push('/order/complete?orderNumber=...')]
```

## 7. ビジネスロジック

### クライアント側処理フロー

1. ページマウント時:
   - `CartContext.items` を読み込み、サマリー描画
   - `items.length === 0` の場合は `/cart` へリダイレクト
2. ユーザーが各フィールドに入力
3. （任意）blur 時に個別バリデーション
4. 「注文する」クリック:
   1. 全項目バリデーション、エラーがあれば送信中止
   2. ボタンを無効化
   3. `items` を `{ book_id: id, title, price, quantity }` 形式に変換
   4. `POST /orders` を実行
   5. 成功時: `clearCart()` を呼び出してカートを空に
   6. `router.push('/order/complete?orderNumber=' + order_number)` で遷移
   7. 失敗時: ボタンを再有効化し、エラーメッセージを表示

### サーバー側処理フロー（参考）

1. リクエストボディを検証（`name`, `address`, `email`, `items` の存在）
2. トランザクション開始
3. `total_amount = Σ price × quantity` を計算
4. `order_number = ORD-${Date.now()}` を生成
5. `orders` テーブルへ INSERT
6. 各 item を `order_items` へ INSERT
7. COMMIT し `{ order_number }` を返却
8. エラー時は ROLLBACK して 500 を返却

### メールアドレスバリデーション

- ブラウザ標準の `<input type="email">` バリデーション
- 加えて正規表現での検証（例: `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`）
- HTML5 の `required` 属性も併用

## 8. エラー処理

| エラー条件 | エラーメッセージ | 対処方法 |
|---|---|---|
| 氏名未入力 | 「氏名を入力してください」 | 該当フィールド下に表示、送信中止 |
| 住所未入力 | 「住所を入力してください」 | 同上 |
| メール未入力 | 「メールアドレスを入力してください」 | 同上 |
| メール形式不正 | 「メールアドレスの形式が正しくありません」 | 同上 |
| カートが空 | （リダイレクト） | `/cart` へリダイレクト |
| API通信失敗（ネットワーク） | 「注文の送信に失敗しました。通信環境をご確認のうえ再度お試しください」 | フォーム上部に表示、再送信可 |
| サーバー側400 | 「入力内容に誤りがあります」 | フォーム上部に表示 |
| サーバー側500 | 「サーバーでエラーが発生しました。時間をおいて再度お試しください」 | フォーム上部に表示、再送信可 |
| 二重送信 | （UIで防止） | ボタン無効化、loading 状態管理 |

## 9. 非機能要件

- **パフォーマンス**:
  - フォーム表示: 初回 < 1秒
  - `POST /orders` 応答: 通常 < 1秒
  - 二重送信防止のためボタン無効化を即座に実施
- **セキュリティ**:
  - メールアドレス・氏名・住所などの個人情報は HTTPS（本番環境）で送信
  - XSS対策: React の自動エスケープ
  - CSRF: 同一オリジン前提（CORS 設定済み）。本番では適切な対策が必要（スコープ外）
  - SQLインジェクション: バックエンドは `mysql2` のパラメータバインディング使用済み
  - サーバー側で価格の再計算を行うのが望ましいが、現状は `items[].price` を信頼している（要件としては許容）
- **アクセシビリティ**:
  - 各 `<input>` に対応する `<label>` を設定
  - エラーメッセージは `aria-describedby` で入力欄と関連付け
  - 送信中は `aria-busy="true"` を設定
  - キーボード操作のみで注文完了まで到達可能

## 10. 制約・前提条件

- 決済処理はスコープ外（要件書 3章）。注文確定 = 注文情報の保存のみ。
- 在庫管理スコープ外のため、注文時に在庫数チェックは行わない。
- ログイン機能なしのため、注文者の同一性確認はメールアドレスのみで管理。
- カート内容は `localStorage` から取得。空の場合は遷移を許可しない。
- フロントエンドコンポーネントは `"use client"` ディレクティブが必須。
- スタイルはインラインスタイルのみ使用。
- 注文番号は `ORD-${Date.now()}` 形式で重複可能性は理論上ゼロではないが、人間が同時刻ミリ秒に注文する確率は極低のため許容。
