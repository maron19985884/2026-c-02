# 注文完了表示機能 外部設計書

## 1. 概要
- **機能ID**: F-05
- **機能名**: 注文完了表示機能
- **対象ユーザー**: 注文を確定した直後のユーザー
- **目的・背景**: 注文が正常に受け付けられたことをユーザーに明示し、注文番号を提示することで、後日問い合わせや注文管理ができるようにする。また、購入後の動線として商品一覧へ戻るリンクを提供し、回遊性を確保する。
- **作成日**: 2026-05-05
- **バージョン**: 1.0
- **関連要件**: U-16, U-17, U-18

## 2. 機能概要

注文フォーム（F-04）からの遷移で `/order/complete?orderNumber=ORD-xxxxxxxxxxxxx` というURLでアクセスされる。クエリパラメータ `orderNumber` を取得し、注文完了メッセージとともに画面上に表示する。同画面下部に「商品一覧に戻る」リンクを設置し、ユーザーが容易にトップへ戻れるようにする。

本画面はデータベースへの問い合わせを行わず、クエリパラメータの値をそのまま表示するシンプルな静的ページ的振る舞いをする（クライアントコンポーネント）。

## 3. 画面・UI設計

### 3.1 画面一覧

| 画面名 | URL/パス | 説明 |
|---|---|---|
| 注文完了画面 | `/order/complete?orderNumber={orderNumber}` | 注文受付完了の通知＋注文番号表示 |

### 3.2 画面レイアウト

#### 構成

1. 大きなアイコン（任意・チェックマーク等）
2. 完了メッセージ: 「ご注文ありがとうございました」
3. 補足メッセージ: 「ご注文を受け付けました」
4. 注文番号表示: 「注文番号: ORD-xxxxxxxxxxxxx」
5. 「商品一覧に戻る」リンク（`/`）

#### 表示例

```
       ✓
   ご注文ありがとうございました
   ご注文を受け付けました

   注文番号: ORD-1714900000000

   [ 商品一覧に戻る ]
```

#### スタイル仕様
- ページ背景: `#f3f4f6`
- メッセージカード: `white`、中央寄せ、`box-shadow: 0 2px 8px rgba(0,0,0,0.08)`
- 完了メッセージ: 大きめサイズ、ネイビー `#1e3a8a`
- 注文番号: 等幅フォント想定、太字、コピーしやすい表示
- リンク／ボタン: `#1e3a8a` テキスト or プライマリボタン
- インラインスタイルのみ使用

### 3.3 画面遷移

```
[注文フォーム (/order)]
     │ POST /orders 成功
     ↓
[注文完了 (/order/complete?orderNumber=ORD-xxx)]
     │
     └─ 「商品一覧に戻る」 → [商品一覧 (/)]
```

注: 直接 URL アクセス（`orderNumber` クエリなし）は想定外だが、その場合の挙動は「8. エラー処理」を参照。

### 5.2 新規エンドポイント

## 6. データ設計

### 6.1 使用テーブル

DBアクセスなし。

### 6.2 データフロー

```
[注文フォーム]
   POST /orders 成功 → { order_number }
       ↓
   router.push('/order/complete?orderNumber=' + order_number)
       ↓
[注文完了画面]
   useSearchParams() → orderNumber 取得 → 画面表示
```

## 7. ビジネスロジック

### 表示処理

1. ページマウント時に Next.js の `useSearchParams()`（または `searchParams` プロパティ）から `orderNumber` を取得
2. `orderNumber` が存在する場合: 完了メッセージ＋注文番号を表示
3. `orderNumber` が存在しない／不正な場合: 「8. エラー処理」を参照

### 副次処理（推奨）

- 表示時には `CartContext` の `items` は既に空（注文確定時に `clearCart()` 済み）。本画面で再度 `clearCart()` を呼ぶことで二重実行を許容することも可能（冪等）。
- ブラウザ「戻る」ボタンで F-04 に戻ると、カートが空のためフォーム画面が再レンダされた瞬間に `/cart` へリダイレクトされる挙動が想定される（F-04 のリダイレクト仕様による）。

## 8. エラー処理

| エラー条件 | エラーメッセージ／挙動 | 対処方法 |
|---|---|---|
| `orderNumber` クエリなし | 「注文情報が見つかりません」表示 | 「商品一覧に戻る」リンクのみ提示。または `/` へリダイレクト |
| `orderNumber` が想定形式外 | 上記と同じ扱い | 同上 |
| ブラウザの戻るで再表示 | 同じ画面が再表示される | 特別な処理は不要（注文番号はURLに残るため再表示可能） |

注: 注文番号の検証はあくまで表示形式上のチェックに留める（DBに照会しない）。要件 U-17「注文番号が画面上に表示されること」を満たせばよい。

## 9. 非機能要件

- **パフォーマンス**:
  - 表示: クライアント側のみで完結するため即時（< 500ms）
- **セキュリティ**:
  - 注文番号はクエリパラメータに露出するが、ブラウザ履歴・referrer 等から漏洩しても直接的な被害はない（注文詳細はDB側にあり、本機能ではDB照会しないため）
  - XSS対策: React の自動エスケープ。注文番号を `dangerouslySetInnerHTML` で扱わないこと
- **アクセシビリティ**:
  - 完了メッセージは `<h1>` にし、スクリーンリーダーでページの意図が伝わるようにする
  - 「商品一覧に戻る」リンクはキーボード操作可能（Tab + Enter）
  - 注文番号は選択・コピーしやすいよう `user-select: text` を維持

## 10. 制約・前提条件

- 注文番号のフォーマットは `ORD-${Date.now()}`（13桁数値が末尾）。
- 本画面は注文情報の取得・参照API を持たない。注文番号の検証は形式チェックのみ。
- ユーザーが意図的に URL を編集して任意の注文番号を表示することは可能だが、それによる被害はない（情報漏洩なし）。
- フロントエンドコンポーネントは `"use client"` ディレクティブが必須（`useSearchParams` 使用）。
- Next.js 14 の App Router 使用前提。`useSearchParams` を使用する場合は `<Suspense>` 境界が必要。
- スタイルはインラインスタイルのみ使用。
