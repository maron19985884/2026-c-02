CLAUDE.md ベストプラクティス

基本原則

CLAUDE.md はすべての会話の開始時に自動で読み込まれる。
Claude への「常時有効な指示書」として機能するため、内容の質と量が直接作業精度に影響する。

書き方のルール

命令形で書く

# 悪い例（説明）
TypeScriptを使用しています。

# 良い例（指示）
TypeScript を使用すること。`any` 型は使わないこと。

Claude は説明より指示に従いやすい。

簡潔に保つ

  • 長すぎると重要な指示が埋もれる
  • 目安は 200〜400行以内
  • コードで確認できることは書かない（ライブラリのAPI仕様など）

重要なことを上に書く

スコープ制約・禁止事項など「絶対に守ってほしいこと」を冒頭に配置する。Claude はファイルの上から読む。

含めるべき内容

項目理由
スコープ制約（やらないこと）勝手な機能追加を防ぐ
テスト実行コマンド変更後に必ず実行させるため
コーディング規約（非自明なもの）コードから読み取れないルールのみ
ディレクトリ構成のルールファイル配置ミスを防ぐ
環境固有のコマンドdocker compose up など

含めないべき内容

項目理由
コードを読めばわかること冗長になるだけ
README と重複する内容二重管理になり陳腐化する
git log / blame で分かること履歴はgitで確認できる
将来の計画・TODOコードに反映されてから書く

階層構造の活用

CLAUDE.md はディレクトリごとに置ける。ルールのスコープを絞れる。

プロジェクトroot/
├── CLAUDE.md          # プロジェクト全体のルール
├── frontend/
│   └── CLAUDE.md      # フロントエンド固有のルール
└── backend/
    └── CLAUDE.md      # バックエンド固有のルール

サブディレクトリの CLAUDE.md はルートのものを引き継いだうえで上書き・追記される。

陳腐化を防ぐ

  • 実装が変わったら CLAUDE.md も同時に更新する
  • 「現在の状態」を書く。「将来こうする予定」は書かない
  • PR レビューの対象に CLAUDE.md を含める
