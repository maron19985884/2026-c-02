# Research: 注文フォームと注文確定処理

**Feature**: 003-order-checkout
**Date**: 2026-06-28

## 1. フォームバリデーション戦略

**Decision**: クライアントサイドバリデーション（フロントエンド）＋ サーバーサイドバリデーション（バックエンド）の二重チェック

**Rationale**:
- クライアントサイド: 即時フィードバックによるUX向上（送信前にエラーを提示）
- サーバーサイド: 信頼できる入力保証（ブラウザ外からの直接APIリクエストにも対応）

**Alternatives considered**:
- クライアントのみ: APIへの直接リクエストに対して脆弱
- サーバーのみ: 送信するまでエラーがわからず、UXが低下する

## 2. 注文番号採番方式

**Decision**: バックエンドがDBへのINSERT時に採番し、レスポンスとしてフロントエンドに返す

**Rationale**:
- 一意性保証をDBに委ねることで重複リスクがゼロになる
- MySQLのAUTO_INCREMENTをベースに `ORD-{YYYYMMDD}-{ID}` 形式の注文番号を生成する
- フロントエンドでは採番せず、確定後のレスポンスのみを使用する

**Alternatives considered**:
- フロントエンドで採番: 重複リスクがありデータ不整合につながる
- UUID v4: 一意性は高いがユーザーに見せる注文番号として長すぎる

## 3. 二重送信防止

**Decision**: Post/Redirect/Get（PRGパターン）を採用

**Rationale**:
- 注文確定後、フロントエンドは `/order/complete` へリダイレクト
- ブラウザのリロードや戻るボタン操作で注文が再送信されない
- 注文番号はリダイレクト時にクエリパラメータまたはNext.jsのrouterで渡す

**Alternatives considered**:
- 同一ページで完了状態を表示: ブラウザリロードで再送信リスクが残る
- 冪等性トークン: 実装コストが高く、学習用途には過剰

## 4. カート情報の取得（注文フォーム画面）

**Decision**: フロントエンドのセッション管理（Context/State）からカート情報を読み込む

**Rationale**:
- カートはセッション単位で管理（spec.md・constitution.md 定義済み）
- 注文フォーム画面ではフロントエンドの状態管理からカート情報を参照する
- カートが空の場合は即座にカート画面へリダイレクトする

**Alternatives considered**:
- バックエンドのカートAPIから再取得: カートがフロントエンド管理である場合は不要であり、実装が複雑になる
