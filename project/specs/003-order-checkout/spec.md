# Feature Specification: 注文フォームと注文確定処理

**Feature Branch**: `feature/003-order-checkout`

**Created**: 2026-06-28

**Status**: Draft

**Input**: User description: "注文フォームと注文確定処理（バリデーション含む）"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 注文フォームにカート内容を確認しながら注文情報を入力する (Priority: P1)

購入希望者が注文フォーム画面を開くと、カートに入っている書籍の一覧と合計金額が同一画面に
表示される。購入者はその内容を確認しながら氏名・住所・メールアドレスを入力できる。

**Why this priority**: 注文フォームの表示と入力が機能しなければ、バリデーションも注文確定も
実行できない。すべての後続ストーリーの前提となる。

**Independent Test**: カートに書籍を入れた状態で注文フォームを開き、フォームと注文内容が
同一画面に表示されることを確認する。入力フィールドに文字を入力できることを確認する。

**Acceptance Scenarios**:

1. **Given** カートに書籍が入っている状態で、**When** 注文フォームページを開くと、**Then** カート内の書名・単価・数量・小計、および合計金額が同一画面に表示される
2. **Given** 注文フォームが表示されている状態で、**When** 氏名・住所・メールアドレスの入力欄を確認すると、**Then** 3つすべての入力フィールドが存在し、入力可能な状態になっている
3. **Given** カートが空の状態で、**When** 注文フォームのページへアクセスすると、**Then** 注文フォームは表示されず、カート画面へリダイレクトされる

---

### User Story 2 - バリデーションエラーを確認して修正する (Priority: P2)

購入希望者が注文フォームで必須項目の入力を忘れたり、メールアドレスの形式が間違っている状態で
「注文する」を押すと、エラーメッセージが表示されて注文が中断される。
エラーを修正して再度送信できる。

**Why this priority**: 不正な入力で注文が通過することを防ぐ重要な品質ゲート。
P1（フォーム表示）が完成した後に単独でテストできる。

**Independent Test**: 注文フォームで意図的に未入力・形式不正の状態で「注文する」を押し、
エラーメッセージが表示され注文が処理されないことを確認する。

**Acceptance Scenarios**:

1. **Given** 氏名が未入力の状態で、**When** 「注文する」ボタンを押すと、**Then** 氏名フィールドの近くにエラーメッセージが表示され、注文処理は実行されない
2. **Given** 住所が未入力の状態で、**When** 「注文する」ボタンを押すと、**Then** 住所フィールドの近くにエラーメッセージが表示され、注文処理は実行されない
3. **Given** メールアドレスが未入力の状態で、**When** 「注文する」ボタンを押すと、**Then** メールアドレスフィールドの近くにエラーメッセージが表示され、注文処理は実行されない
4. **Given** メールアドレスに「@」を含まない文字列が入力されている状態で、**When** 「注文する」ボタンを押すと、**Then** 「メールアドレスの形式が正しくありません」というエラーメッセージが表示され、注文処理は実行されない
5. **Given** 複数の項目が未入力の状態で、**When** 「注文する」ボタンを押すと、**Then** 未入力のすべての項目にそれぞれエラーメッセージが表示される
6. **Given** エラーが表示されている状態で、**When** エラーを修正して「注文する」を再度押すと、**Then** エラーが解消され注文処理が続行される

---

### User Story 3 - 注文を確定して注文番号を受け取る (Priority: P3)

購入希望者が正しく入力した注文フォームで「注文する」ボタンを押すと注文が確定し、
注文完了画面で一意の注文番号が表示される。商品一覧へ戻ることができる。

**Why this priority**: P1・P2が完成した後に実現できる最終ゴール。
注文確定処理と完了画面は注文フォームの完成形。

**Independent Test**: 正しい情報を入力した状態で「注文する」を押し、
注文完了画面と注文番号が表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** 全項目が正しく入力されている状態で、**When** 「注文する」ボタンを押すと、**Then** 注文が確定し注文完了画面へ遷移する
2. **Given** 注文完了画面が表示された状態で、**When** 画面を確認すると、**Then** 注文完了を知らせるメッセージと一意の注文番号が表示されている
3. **Given** 注文完了画面が表示された状態で、**When** 「商品一覧へ戻る」リンクをクリックすると、**Then** 商品一覧ページへ遷移する
4. **Given** 複数の注文を個別に確定した場合、**When** それぞれの注文番号を確認すると、**Then** すべての注文番号が異なる（重複なし）

---

### Edge Cases

- カートが空の状態で注文フォームへ直接アクセスした場合、カート画面へリダイレクトする
- 全必須項目が未入力の場合、すべての項目にエラーメッセージを同時に表示する
- メールアドレスが「@」を含まない場合、形式不正エラーを表示する
- 注文確定後にブラウザの戻るボタンで注文フォームに戻っても、同一注文が二重に送信されない

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 注文フォーム画面には入力フィールドと同一画面にカート内の書名・単価・数量・小計および合計金額を表示しなければならない
- **FR-002**: 注文フォームには氏名・住所・メールアドレスの3項目の入力フィールドを提供しなければならない（全項目必須）
- **FR-003**: 「注文する」ボタンを押したとき、未入力の必須項目がある場合は該当フィールドにエラーメッセージを表示しなければならない
- **FR-004**: メールアドレスの形式が不正な場合、「メールアドレスの形式が正しくありません」というエラーメッセージを表示しなければならない
- **FR-005**: バリデーションエラーが1つでも存在する場合、注文確定処理を実行してはならない
- **FR-006**: 全バリデーションを通過した状態で「注文する」ボタンを押すと、注文を確定しなければならない
- **FR-007**: 注文確定時にシステムは一意の注文番号を発行しなければならない
- **FR-008**: 注文完了画面には注文完了メッセージと注文番号を表示しなければならない
- **FR-009**: 注文完了画面には商品一覧へ戻るリンクを設けなければならない
- **FR-010**: カートが空の状態で注文フォームへアクセスした場合、カート画面へリダイレクトしなければならない

### Key Entities *(include if feature involves data)*

- **注文入力情報 (OrderInput)**: ユーザーがフォームに入力する情報。氏名・住所・メールアドレスを持つ
- **注文 (Order)**: 確定された購入記録。一意の注文番号・注文入力情報・注文日時・注文アイテムリストを持つ
- **注文アイテム (OrderItem)**: 注文に含まれる書籍の記録。書籍・数量・注文時単価を保持する（価格変動の影響を受けない）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 全項目を正しく入力して「注文する」を押してから注文完了画面が表示されるまでが3秒以内に完了する
- **SC-002**: 必須項目が未入力の場合、100%のケースでエラーメッセージが表示され注文が中断される
- **SC-003**: メールアドレスの形式が不正な場合、100%のケースで形式エラーメッセージが表示され注文が中断される
- **SC-004**: 注文確定のたびに一意の注文番号が発行され、重複が発生しない
- **SC-005**: 注文フォーム画面の初期表示が3秒以内に完了する

## Assumptions

- ユーザー認証は実装しない（ゲスト注文のみ）
- 決済処理は実装しない（注文情報の記録のみ行う）
- 注文番号はバックエンドが採番してフロントエンドに返す
- メールアドレスの形式チェックは「`@`を含む標準的なメール形式」を基準とする
- 注文確定後にカートの中身をクリアするかどうかは、購買フロー全体仕様（001-purchase-flow）に従う
- 注文履歴の参照・注文キャンセルはスコープ外
- 注文確認メールの送信はスコープ外
- 注文アイテムの単価は注文確定時点の価格を記録する（後日の価格変動の影響を受けない）
