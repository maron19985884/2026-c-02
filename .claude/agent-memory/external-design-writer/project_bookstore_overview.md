このプロジェクト（個人運営オンライン書店・購買フロー特化版）の外部設計書を作成する際、要件書には明示されていないが下記の判断・解釈を採用した。

**Why:** 要件書 `user_requirements.md`（注: ファイル名は複数形 `requirements`、単数形 `requirement` ではない）はユーザー視点の "したいこと" のみを書いているため、技術的な制約・実装方針は CLAUDE.md と既存コードから推論する必要がある。

**How to apply:** 同じプロジェクトの設計書を更新・追加する場合は、以下の解釈を維持して整合性を保つこと。

## 判断した解釈

1. **カートの永続化方針**: `CartContext` は `localStorage` のみで永続化する設計（要件書には書かれていない）。サーバー側にカート状態は保存しない。`Providers.tsx` 経由でアプリ全体をラップ。

2. **注文番号フォーマット**: `ORD-${Date.now()}` 形式。`backend/src/index.ts` 実装に依拠。理論上の重複可能性はあるが許容（人間が同時刻ミリ秒に注文する確率が極低のため）。

3. **注文完了画面の注文番号取得**: クエリパラメータ `?orderNumber=` から取得し、DBへの問い合わせは行わない。`useSearchParams` 使用のため `<Suspense>` 境界が必要（Next.js 14 App Router の制約）。

4. **価格の信頼境界**: バックエンド `POST /orders` は `items[].price` をそのまま信頼している（再計算しない）。本来はサーバー側で `books` テーブルから引き直すべきだが、現状実装では許容。

5. **空カート時のフォーム遷移防止**: 要件書には書かれていないが、`/order` への直接アクセスでカートが空の場合は `/cart` へリダイレクトする運用前提とする。

6. **エンドポイント数**: バックエンドは `/health, /products, /products/:id, /orders` の4本のみ。すべて `backend/src/index.ts` に集約。新規エンドポイント追加が必要な機能要件は現時点でない。

7. **画面 = 機能の単位**: 要件書が画面ごとに整理されているため、外部設計書も画面 = 機能（F-01〜F-05）の単位で1ファイルずつ作成する設計方針を採用した。
