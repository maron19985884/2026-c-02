# ウォーターフォール開発における Claude Code 活用ガイド

## 基本的な考え方

ウォーターフォールは「各フェーズの成果物（ドキュメント）が次のフェーズへのインプット」になる構造です。
Claude Code はその **成果物の生成・レビュー・変換** で力を発揮します。

---

## フェーズごとの活用方法

### ① 要件定義

- 要件ドキュメントのレビュー・矛盾チェック
- ユーザーストーリーの整理、受け入れ条件のドラフト生成
- 既存コードベースを読み込ませて「現状」と「要件」のギャップ分析

### ② 基本設計

- アーキテクチャ案の作成、設計書のたたき台生成
- DB 設計・API 設計のレビューと改善提案
- クラス図・シーケンス図の Mermaid ソース生成

### ③ 詳細設計

- 関数・クラスレベルの仕様書テンプレート生成
- インターフェース定義（OpenAPI / TypeScript 型）の自動生成
- 設計書とコードの整合性チェック（既存実装を渡してレビュー）

### ④ 実装

- 設計書を渡してボイラープレート・スキャフォールドを自動生成
  - スキャフォールド = モデル・API エンドポイント・テストひな形など「骨組み」一式の生成
- コードレビュー（バグ検出・リファクタリング提案）
- 難解ロジックへのコメント・ドキュメント自動挿入

### ⑤ テスト

- テスト仕様書から単体・結合テストコードを自動生成
- テスト観点の網羅チェック（境界値・異常系の提案）
- バグ報告文を渡して原因調査・修正パッチ生成

### ⑥ リリース・保守

- リリースノート・変更履歴の自動生成
- 運用手順書・障害対応マニュアルのドラフト作成
- 保守時の影響範囲調査（コードベース検索・依存関係分析）

---

## 運用のコツ

| ポイント | 内容 |
|---|---|
| 成果物を必ず渡す | 前フェーズのドキュメントをコンテキストとして与えると、フェーズ間の一貫性が保たれる |
| 人間がゲートキーパーになる | 各フェーズの承認ポイントで Claude の出力を人間がレビュー・承認する |
| CLAUDE.md を活用する | コーディング規約や設計方針を記載しておくと実装フェーズの出力品質が安定する |

## 用語メモ

**スキャフォールド（scaffold）**
直訳は「足場」。モデル・APIエンドポイント・テストファイルなど、アプリケーションの骨組みとなるコード一式を自動生成すること。
中身のビジネスロジックは空の状態だが、ファイル構成・クラス名・メソッド名・インターフェースが揃った状態からスタートできる。
