# ブランチ比較レポート（画面の振る舞い・ユーザー体験）

---

## はじめに

このレポートは、オンライン書店アプリを実装した5つのブランチ（開発の枝分かれ）を、「どんな画面があり、何ができるのか」という観点から比較したものです。

ブランチとは、複数の開発者が同じプロジェクトに対してそれぞれ独立して作業するための仕組みです。5人の開発者がそれぞれ別々に購買フロー（商品一覧 → 商品詳細 → カート → 注文フォーム → 注文完了）を実装しました。

**比較対象の5ブランチ：**
- `AkaneOno` / `h1nakamu`：未実装ブランチ（アプリの骨組みのみ）
- `kazukisugawara2` / `keitaKobayashi` / `satoyuya`：購買フローを全画面実装済みのブランチ

技術に詳しくない方向けに、コードの詳細ではなく「実際に使ったときの体験がどう違うか」に焦点を当てて記述しています。

## 2. 画面ごとの比較

### 2-1. 商品一覧画面

#### 共通している点（kazukisugawara2 / keitaKobayashi / satoyuya）
- 書籍のタイトル・著者・価格が表示される
- 書籍をグリッド（複数列）で並べて表示する
- 書籍をクリックすると詳細画面へ移動できる
- 常に最新のデータベース情報を表示する（キャッシュを使わない、または定期的に更新する）

#### ブランチ別の違いとユーザーへの影響

| ブランチ | 動作・見た目 | ユーザーへの影響 |
|---------|-------------|----------------|
| kazukisugawara2 | ページを開いた瞬間から書籍一覧が表示される（サーバー側でデータを取得してから画面を届ける仕組み）。データベースに書影（表紙画像）URLが登録されていれば実際の画像を表示し、未登録なら「No Image」と表示する | 「読み込み中...」の待ち時間がなく、画面がちらつかない。表示が安定していて快適 |
| keitaKobayashi | ページを開くと一瞬「読み込み中...」が表示され、その後に書籍一覧が出る（ブラウザ側でデータを取得する仕組み）。書影は外部のプレースホルダー画像サービスを利用して必ず何らかの画像を表示する。マウスを乗せるとカードが少し浮き上がるアニメーションがある | 最初に一瞬「読み込み中...」が見えるが、すぐに一覧が出る。どの書籍にも画像があるのでビジュアルとして見やすい。ホバー効果で操作感がある |
| satoyuya | ページを開くと一瞬「読み込み中...」が表示され、その後に書籍一覧が出る。書影URLがなくても、書籍タイトルの文字から自動で色を決めてカラフルな擬似書影（表紙イメージ）を生成して表示する | 最初に「読み込み中...」が見える。書影URLがなくても見た目が揃っており、書店らしい雰囲気がある。統一感のあるデザイン |

### 2-3. カート画面

#### 共通している点（kazukisugawara2 / keitaKobayashi / satoyuya）
- 商品名・単価・数量・小計を表示する
- 数量の増減ボタン（＋・−）がある
- 商品の削除ができる
- 合計金額が表示される
- カートが空の場合は「空です」旨を表示する
- 「注文手続きへ」ボタンがある

#### ブランチ別の違いとユーザーへの影響

| ブランチ | 動作・見た目 | ユーザーへの影響 |
|---------|-------------|----------------|
| kazukisugawara2 | 数量「−」ボタンは数量が1のとき押せなくなる（グレーアウト）。カートが空のとき「カートに商品がありません」と表示され「書籍一覧へ戻る」リンクが出る。カートが空の場合、注文フォームへのリンクは表示されない。ページ初回表示時に一瞬「読み込み中...」が出る（ブラウザのローカルストレージ（端末内保存領域）からデータを読む際の処理） | 誤って数量を0以下にできない。空のカートから間違えて注文フォームに進めない設計 |
| keitaKobayashi | 数量「−」ボタンは数量が1でも押せる（`Math.max(1, quantity-1)` の処理があるため実際には1未満にはならないが、ボタン自体はグレーアウトしない）。カートの各商品行に書影（プレースホルダー画像）が表示される。カートが空でも「注文手続きへ」ボタンは存在しない（空の場合の専用画面を表示）。合計金額はリアルタイムで更新される | 画像があって商品が視覚的に確認しやすい。カートが空のときに注文フォームへは進めない |
| satoyuya | 数量「−」ボタンを押すと数量が0以下になると商品がカートから自動で削除される（cartReducerのロジック）。カート画面が左右2カラム（商品一覧 ＋ 注文サマリー）のレイアウトで、注文サマリーが右側に固定表示される。書影は擬似書影（自動生成）が表示される。合計金額とともに「商品点数」もサマリーに表示される | 数量0にすると自動削除される点は直感的な反面、意図せず削除されることもある。サマリーが常に見えていて合計を確認しやすい |

### 2-4. 注文フォーム画面

#### 共通している点（kazukisugawara2 / keitaKobayashi / satoyuya）
- 氏名・住所・メールアドレスの入力欄がある
- 必須項目の入力チェック（バリデーション）がある
- メールアドレスの形式チェック（@が含まれているかなど）がある
- エラーの場合は赤色でエラーメッセージを表示する
- 注文する前に注文内容（カートの中身）を同じ画面で確認できる
- 送信中はボタンを押せなくなる（二重送信の防止）
- 注文確定後にカートの中身が空になる

#### ブランチ別の違いとユーザーへの影響

| ブランチ | 動作・見た目 | ユーザーへの影響 |
|---------|-------------|----------------|
| kazukisugawara2 | バリデーション（入力チェック）がフォーカスを外したとき（入力欄から離れたとき）と送信ボタンを押したときの2タイミングで走る。入力中はエラーがクリアされる。文字数上限のチェックもある（氏名100文字、住所255文字など）。カートが空の場合は自動的にカート画面にリダイレクト（転送）される。注文内容確認エリアが画面上部に表示され、その下にフォームが続く縦並びのレイアウト。住所は1行の入力欄 | フォーカスを外すとその場でエラーがわかり、送信前に修正しやすい。カートが空の状態で注文フォームを直接開いてもカートに戻される |
| keitaKobayashi | バリデーションは送信ボタンを押したときのみ走る。入力中・フォーカスを外したときにはチェックしない。注文フォーム（左）と注文内容確認（右）が横並び2カラムのレイアウト。住所は1行の入力欄。カートが空の場合は「カートが空のため注文できません」と表示してフォームは表示しない | 入力途中でエラーは出ない。送信してはじめてエラーがわかる。注文内容と入力欄を並べて確認しながら入力できる |

## 3. 操作フロー全体を通じた比較

### kazukisugawara2 ブランチの購買体験

最も「ちらつき」が少なく快適な体験を提供します。商品一覧・詳細ページはサーバー側でデータを用意してから画面を届ける仕組み（Server Component）を使っているため、「読み込み中...」という表示が出ません。ページを開いた瞬間から情報が揃っています。

カート管理は端末内のデータ（localStorage）を使うため、ブラウザを閉じて再度開いてもカートの中身が残ります。注文フォームでは入力欄からフォーカスが外れた時点でその項目だけエラーチェックが走るため、送信前に間違いに気づいて修正できます。

注文完了画面はシンプルで、注文番号のみが大きく表示されます。お届け先や明細の確認はできませんが、注文番号はURLに含まれているため、後から同じ画面を開けば番号を確認できます。

全体として、画面遷移ごとにデータ取得タイミングを適切に設計しており、フロー全体を通じて一貫して使いやすい体験になっています。

### keitaKobayashi ブランチの購買体験

全画面に渡ってビジュアル面に力が入っています。書籍一覧ではカードにマウスを乗せると浮き上がるアニメーションがあり、プレースホルダー画像によってどの書籍も必ずサムネイルが表示されます。カート画面でも書籍のサムネイルが表示されており、「何を買おうとしているか」が視覚的にわかります。

注文フォームは入力欄（左）と注文内容確認（右）が横並びのレイアウトで、入力しながら注文内容を確認できる使いやすさがあります。ただし、バリデーション（入力チェック）は送信ボタンを押したときのみ動作するため、記入ミスは送信ボタンを押してはじめてわかります。

注文完了画面は最も情報量が多く、名前・メールアドレス・お届け先・注文書籍一覧・合計金額という「確認書」のような内容が一画面に揃います。

### satoyuya ブランチの購買体験

デザインに独自性があります。書影（表紙画像）が存在しない場合でも、書籍タイトルの文字情報を元に自動でカラフルな擬似書影を生成するため、画像なしの書籍でも見た目が揃っています。ダーク系のカラーパレット（紺・黒・えんじ色など）で統一されており、スタイリッシュな印象の書店になっています。

カート画面は左右2カラムで、右側に「注文サマリー」が常に固定表示されます。スクロールしても合計金額と「注文手続きへ」ボタンが常に見えている点が便利です。

注文完了画面は注文番号・日時・お届け先情報・注文明細を全て表示する最も詳細な画面です。注文日時も日本語フォーマットで表示されるため、「いつ注文したか」もその場で確認できます。

カート管理の実装（Reducer（更新ロジックを一箇所にまとめる仕組み））が3ブランチの中で最も整理されていますが、数量を0に変更すると商品が自動削除される仕様のため、意図せず削除してしまう可能性があります。

## 4. まとめ

各ブランチをひとことで特徴づけると以下のとおりです。

| ブランチ | ひとことの特徴 |
|---------|--------------|
| **AkaneOno** | 要件定義書のみ存在。アプリとして動く状態ではない。 |
| **h1nakamu** | 環境構築のスケルトン（骨組み）のみ。書店機能は未実装。 |
| **kazukisugawara2** | 「ちらつきのない快適な表示」と「入力途中でのエラー検知」を両立した、最もスムーズな購買体験。 |
| **keitaKobayashi** | 画像やアニメーションによる「見た目の充実」と「完了後の詳細確認」を重視したビジュアル重視の実装。 |
| **satoyuya** | 独自の擬似書影生成とスタイリッシュなデザインで統一感があり、注文完了後の情報量も最も豊富な実装。 |

実装済みの3ブランチはいずれも購買フロー全体（商品一覧→詳細→カート→注文フォーム→注文完了）が動作しており、書籍を選んで注文を完了させることができます。ユーザー体験の方向性が異なるため、どのブランチが優れているかは「何を重視するか」によって変わります。

- 画面表示の快適さ・入力エラーの早期検知を優先するなら：**kazukisugawara2**
- 視覚的な豊かさ・完了後の注文確認しやすさを優先するなら：**keitaKobayashi**
- デザインの統一感・注文完了後の情報の詳しさを優先するなら：**satoyuya**
