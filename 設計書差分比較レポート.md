### feature/AkaneOno — 外部設計（画面別）＋詳細設計（機能別）＋システム設計書

ファイル構成:
- `docs/external_design/` — 5ファイル（画面1枚ごとに独立）
- `docs/detailed_design/` — 7ファイル（API・コンポーネント・データモデル別）
- `docs/system_design.md` — システム設計書（データモデル・API・ディレクトリ構成）

#### 設計書の特徴

- **最もファイル数が多い（13ファイル）**。画面1枚・API1本ごとに独立ファイルで管理
- 外部設計書はワイヤーフレーム（ASCIIアート）・表示条件・状態遷移を含む詳細な記述
- `system_design.md` はTypeScript型定義・バリデーションルール・処理仕様を網羅した実装指向の設計書

#### データモデル

| エンティティ | 型 | 永続化 |
|---|---|---|
| `Book` | TypeScript interface（UUID id） | `backend/src/data/books.json` |
| `Order` | TypeScript interface | `backend/src/data/orders.json` |
| `OrderItem` | `Order` に内包 | 同上（スナップショット: title・price・subtotal） |
| `CartItem` | TypeScript interface | localStorage |

> **他ブランチと異なる点:** 書籍IDが **UUID**（他ブランチはINT AUTO_INCREMENT）。データ永続化が **JSONファイル**（他ブランチはMySQL）。

#### 画面・URL一覧

| 画面 | URL |
|---|---|
| 商品一覧 | `/` |
| 商品詳細 | `/books/[id]` |
| カート | `/cart` |
| 注文フォーム | `/checkout` |
| 注文完了 | `/order-complete` |

#### API設計

| メソッド | パス | バックエンドポート |
|---|---|---|
| GET | `/api/books` | **3001**（他ブランチは4000） |
| GET | `/api/books/:id` | 3001 |
| POST | `/api/orders` | 3001 |

- 書籍一覧: Next.js サーバーコンポーネントでフェッチ（ISR `revalidate=60`）
- 注文完了画面: `POST /api/orders` のレスポンスを router.push + 状態経由で引き渡す（注文完了画面からの再取得APIなし）
- `unit_price` はクライアントから送信せず、**サーバー側で書籍データから取得して計算**

#### 注文番号

| 形式 | 例 | 採番方式 |
|---|---|---|
| `ORD-YYYYMMDD-NNNNNN` | `ORD-20260527-000001` | `orders.json` の全件数+1の6桁ゼロ埋め |

**特徴:**
- データ永続化をJSONファイルで行うシンプルなアーキテクチャ
- TypeScript型定義を `system_design.md` に集中管理し、実装との対応が明確
- 外部設計書にワイヤーフレームを記述（UI設計の意図が読み取りやすい）
- バックエンドポートが **3001**（他ブランチは4000）
- `OrderItem` に `subtotal` カラムを含む（他ブランチは含まないものが多い）

### feature/kazukisugawara2 — 外部設計書（What）＋内部設計書（How）の2分割構成

ファイル構成:
- `external_design.md` — 外部設計書（画面・API・データモデル定義）
- `internal_design.md` — 内部設計書（実装レベルの詳細設計）

#### 設計書の特徴

- **What（外部）と How（内部）を明確に役割分離**した2ドキュメント構成
- `internal_design.md` の冒頭に「外部設計書が『何を作るか』を定義し、本書は『どのように作るか』を定義する」と明記
- DDL・TypeScript型定義・APIクライアントのコードサンプルまで含む実装指向の内部設計書

#### データモデル・テーブル設計

| テーブル | 主なカラム | 備考 |
|---|---|---|
| `products` | id, title, author, price, image_url, description, created_at, updated_at | **他ブランチは`books`** |
| `orders` | id, order_number, customer_name, customer_address, customer_email, total_amount, created_at, updated_at | `updated_at` 有（他ブランチはなし） |
| `order_items` | id, order_id, product_id, title, price, quantity, subtotal | **title・subtotalのスナップショット**あり、**インデックス定義**あり |

> **他ブランチと異なる点:** テーブル名・APIパスが `products`（他3ブランチは `books`）。`order_items` にインデックス（`idx_order_items_order_id` / `idx_order_items_product_id`）を明示定義。

#### 画面・URL一覧

| 画面 | URL |
|---|---|
| 商品一覧 | `/` |
| 商品詳細 | `/products/[id]` |
| カート | `/cart` |
| 注文フォーム | `/order` |
| 注文完了 | `/order/complete` |

> **注文完了URLにIDを含めない**唯一のブランチ。直接URLでの注文確認ページアクセスは不可。

#### API設計

| メソッド | パス | 備考 |
|---|---|---|
| GET | `/api/products` | 書籍一覧（`cache: 'no-store'` で常に最新） |
| GET | `/api/products/:id` | 書籍詳細 |
| POST | `/api/orders` | `unit_price` はサーバー側でDB参照 |
| GET | `/api/orders/:id` | 注文詳細 |

#### 注文番号

| 形式 | 例 | 採番方式 |
|---|---|---|
| `ORD-0000000001` | `ORD-0000000001` | `orders.insertId` の10桁ゼロ埋め |

**特徴:**
- 設計書を「読む対象（PO/PM）」と「実装する対象（開発者）」で分離した構成
- コンポーネント粒度が最も細かい（`CartItemRow`・`OrderSummary`・`ErrorMessage` を明示）
- キャッシュ戦略を `no-store` で明記（常に最新データを優先）
- 注文番号の連番形式はDBの `insertId` に依存するシンプルな設計
